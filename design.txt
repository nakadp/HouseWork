📝 家务管理 Line Mini App 产品设计书
一、 整体架构与导航设计

应用采用底部全局导航栏（Tab Bar），保持与参考软件一致的深色/浅色沉浸式界面风格，突出核心功能。

底部导航栏规划（从左至右）：

    主页 (对应原“训练”)：展示用户本人的今日家务及预设家务模板。

    家庭 (对应原“挑战”)：家庭成员的动态广场，查看所有人完成的家务。

    家务库 (对应原中间大按钮/“动作”)：核心入口，按区域分类添加家务。

    历史 (对应原“历史”)：日历视图与数据统计。

    我的 (对应原“我的”)：个人信息与家庭管理。

二、 核心页面详细设计
1. 首页：今日家务（参考图一）

交互逻辑： 聚焦用户本人的单日任务，提供快捷操作入口。

    顶部看板（今日家务）：

        显示“第X次打扫”或“今日家务”。

        显示摘要：例如“包含动作 5 ｜ 预计耗时 30分钟”。

        右侧醒目的「GO! / 开始」按钮，点击进入依次打卡模式。

    打卡与拍照记录（新增功能）：

        在完成某一项家务时，系统弹出提示或提供一个 📷 图标，支持调用手机相机或相册，上传照片/截图作为完成凭证。

    家务计划/模板专区（官方计划）：

        水平滑动的卡片组，提供预设模板，如：“周末大扫除”、“厨房油污专项”、“十分钟快速整理”、“睡前归位法”。

        分级展示（如日常、深度、专项），点击即可一键将一整套家务导入“今日家务”。

2. 家务库：选择与分类（参考图二）

交互逻辑： 采用左侧一级分类（区域）、右侧瀑布流/网格展示具体项目（家务）的经典双联动布局。

    左侧导航（区域分类）：

        默认列表：客厅、主卧、次卧、厨房、卫生间、阳台等。

        自定义功能：列表最底部或顶部固定一个「➕ 添加区域」按钮，支持用户手动输入自定义位置（如“车库”、“宠物角”）。

    右侧内容（具体家务）：

        图文卡片展示：如“扫地”、“拖地”、“清理油烟机”、“给植物浇水”。

        自定义功能：右上角或列表末尾提供「➕ 自定义家务」，可设定名称、图标并归属到特定区域。

    操作交互：点击某一家务卡片，弹出悬浮窗，可选择“加入今日家务”或“设置提醒频率”。

3. 历史与统计：日历看板（参考图三）

交互逻辑： 直观呈现劳作记录，支持多维度筛选。

    顶部切换：提供「历史（日历）」与「统计（图表）」两个Tab。

    日历视图（历史Tab）：

        完整展示当前月份，有家务记录的日期下方显示彩色高亮条块。

        家庭成员区分（关键定制）：不同家庭成员使用不同的颜色标签（例如：爸爸蓝色、妈妈红色、孩子黄色）。日历上的日期格子会堆叠显示当天的颜色块，一眼看出是谁做了家务。

    详情抽屉：

        点击日历上某一天，下方滑出当日家务清单。

        列表内带有成员头像标识，以及他们上传的拍照截图记录的缩略图，点击可放大查看打扫成果。

    报表与设置：右上角提供数据筛选，可按“成员”或“区域”筛选历史数据。

4. 个人中心与家庭设置（参考图四）

交互逻辑： 集中管理账号属性、偏好与协作关系。

    顶部个人信息卡片：

        用户头像、昵称、修改信息入口。

        家务称号/等级（如“家务小能手”、“金牌管家”，替代原“永久会员”位置，增加趣味性）。

    功能列表区：

        家庭成员管理（核心新增）：

            点击进入后，生成专属的 Line 邀请链接或二维码。

            发送给Line好友，对方点击授权后即可加入当前“家庭组”。

            列表展示当前家庭所有成员，支持管理员移除成员或修改成员备注。

        偏好设置：提醒时间设置、UI主题颜色切换。

        身体数据/照片墙转换：原“照片墙”可改为“家务成果墙”，集中展示以往打卡上传的优秀卫生照片。

        历史备忘录：家电维修记录、耗材购买备忘录（如垃圾袋、清洁剂库存提醒）。

三、 特色协作交互流程设计 (User Flow)

    发起协作 (邀请加入)
    我的 -> 家庭成员管理 -> 邀请微信/Line好友 -> 好友在Line内打开小程序并同意 -> 数据互通，历史记录自动同步。

    执行与记录 (加入照片验证)
    首页 -> 点击 开始今日家务 -> 勾选 洗碗 -> 弹窗提示 是否上传照片记录？ -> 拍照 -> 确认完成。

    成果展示 (家庭公开透明)
    任意成员进入 历史 -> 点击 今天 -> 看到成员A（红色头像）完成了“洗碗”并附带了闪亮的水槽照片 -> 其他成员可点击“点赞”互动。



    补充描述：核心边界条件与系统机制
1. 角色与权限管理机制 (Role & Permission Control)

既然引入了“家庭协作”，就需要明确数据的所有权和修改权。

    管理员 vs. 普通成员：创建家庭组的人自动成为管理员。只有管理员可以踢出成员、修改家庭维度的全局设置（如删除自定义的“区域”）。

    防误触与数据修改：成员A完成了家务并上传了照片，成员B是否可以点击“撤销”或删除该记录？通常建议仅限创建者本人或管理员可以修改/删除历史记录。

2. Line 生态特有功能整合 (Line API Integration)

作为 Line Mini App (LIFF)，我们可以深度利用 Line 的原生能力，这部分需要在设计中体现：

    消息推送 (Messaging API)：当有人加入家庭、或者某人上传了带有照片的“家务打卡”时，可以通过 Line Bot 向家庭群组发送一条卡片消息（例如：“🎉 爸爸刚刚完成了【清理油烟机】，快来看看成果！”），增强互动性。

    快捷分享 (Share Target Picker)：除了生成邀请链接，可以直接唤起 Line 的联系人列表，一键发送精美的家庭邀请卡片。

3. 激励与正反馈闭环 (Gamification)

参考健身软件中“挑战”、“目标”的概念，家务管理很容易让人产生倦怠感。

    积分或勋章系统：结合之前提到的“称号”，可以补充一个轻量级的积分系统。每完成一项家务积攒点数，并在“家庭”Tab 中展示本周的“家务贡献排行榜”，增加趣味性和动力。

4. 离线与并发冲突处理 (Data Synchronization)

当两位家庭成员在超市同时勾选了“购买清洁剂”，或者同时在家清理同一区域并在无网络状态下打卡：

    需要在设计文档中补充状态同步策略。例如前端展示的加载状态（Loading UI），以及当数据发生冲突时，是以服务器时间为准还是保留双份记录。



一、 核心技术栈映射 (Firebase 免费全家桶)

我们将 Line Mini App (LIFF) 的前端与 Firebase 的三大核心服务直接对接：

    实时数据库：Cloud Firestore

        作用： 存储家务模板、家庭成员关系、每日打卡记录。

        魔法特性： 使用 onSnapshot 监听。当家庭成员 A 在他的手机上点击“完成洗碗”，成员 B 的屏幕上会瞬间自动打勾，不需要下拉刷新。

    文件存储：Cloud Storage for Firebase

        作用： 存储用户完成家务后上传的“拍照截图”凭证。

    身份验证：Firebase Authentication

        作用： 结合 Line Login，安全地管理用户身份。由于 LIFF 自带 Line 登录，您可以静默获取用户的 Line UserID，以此作为 Firebase 的唯一身份标识。

二、 极简数据库模型设计 (NoSQL)

在 Firestore 中，数据是以“集合 (Collection)”和“文档 (Document)”的形式组织的。为了实现家庭数据的隔离和共享，建议采用以下结构：

    📁 集合：families (家庭群组)

        📄 文档：{family_id} (比如由邀请链接生成的唯一ID)

            name: "张三的家"

            members: ["line_user_id_1", "line_user_id_2"] (包含家庭所有成员的 Line ID 数组，用于权限控制)

            created_at: 2026-06-02

        ↳ 📂 子集合：chores_templates (家务库/模板)

            📄 文档：{chore_id}

                title: "洗碗"

                area: "厨房"

                icon: "🍽️"

                created_by: "line_user_id_1"

        ↳ 📂 子集合：records (历史打卡记录)

            📄 文档：{record_id}

                chore_title: "洗碗"

                completed_by: "line_user_id_2" (标识是谁做的)

                completed_at: 时间戳 (用于日历和统计)

                photo_url: "https://firebasestorage..." (存储在 Firebase Storage 中的照片链接)

三、 关键业务逻辑的“简单”实现方法
1. 加入家庭 (免后端邀请机制)

    传统做法：需要后端生成邀请码，校验过期时间等。

    Firebase 极简做法：

        管理员在小程序内点击“邀请成员”，前端直接生成一个带有 family_id 的 Line 分享链接（例如：https://liff.line.me/xxx?familyId=12345）。

        被邀请人点击链接打开 LIFF，前端解析出 familyId。

        前端直接调用 Firestore API，将该用户的 Line ID arrayUnion (追加) 到 families/{family_id} 的 members 数组中。完成加入！

2. 上传家务照片

    极简做法：

        调用 Line 提供的相机接口或手机原生 <input type="file" capture="camera"> 拍照。

        前端直接将图片文件上传到 Firebase Storage 的 /{family_id}/photos/ 目录下。

        获取图片的下载 URL (DownloadURL)，连同打卡记录一起写入 records 集合。

3. 首页实时进度板

    在首页，前端直接监听当日的 records 数据：
    JavaScript

    // 伪代码示例：实时监听今天的家务记录
    const todayRecordsRef = query(
        collection(db, `families/${userFamilyId}/records`),
        where("completed_at", ">=", todayStart)
    );

    onSnapshot(todayRecordsRef, (snapshot) => {
        // 只要有人做完家务，这里就会触发，直接更新UI上的进度条和列表
        updateUI(snapshot.docs); 
    });

四、 💡 需要注意的现实问题（避坑指南）

虽然 Firebase 免费且快速，但作为开发者，您需要注意免费额度（Spark Plan 的限制）：

    数据库读取次数 (5万次/天 免费)：

        优化建议：日历历史页（图三）千万不要一次性拉取全年的数据。一定要按“月”查询，只有用户切换月份时再去请求新的数据，否则你的免费读取额度会很快耗尽。

    存储空间 (5 GB 免费)：

        优化建议：家务打卡的照片不需要高清大图。在前端上传至 Firebase 之前，务必使用 JavaScript (如 browser-image-compression 库) 将图片压缩到 100-200KB 左右。这样 5GB 可以存几万张照片，完全够免费用很久。

    数据安全 (Security Rules)：

        因为没有后端，千万不要在控制台把 Firebase 权限设置为对所有人读写（allow read, write: if true;）。您只需要写一行简单的规则：只允许用户的 Line ID 存在于 members 数组中时，才能读写该家庭的数据。

采用这套方案，您甚至不需要购买服务器，只需懂前端（HTML/CSS/JS 或 Vue/React）配合 Line LIFF SDK 就可以把整个产品的核心逻辑跑通。
